# Digital Company 产品规划 & 技术设计评审

这份文档整体质量相当高，尤其是产品定义部分展示了少见的自我克制——能在 v0.2 就把"它不是什么"写得这么清楚，说明产品思维足够成熟。技术方案在架构选型上也做了合理的收敛。以下按维度逐一展开。

---

## 一、产品定义评审

### **核心定义的清晰度：强**

"cadence-driven operating product vs prompt-driven AI tool"这个区分是这份文档最有价值的一句话，它直接戳穿了市场上绝大多数竞品的本质。从当前竞品格局来看，CrewAI 是 role-based 的多 agent 协作框架，AutoGen 是 conversational multi-agent，Lindy.ai 是 no-code AI agent 平台，Relay.app 是 AI-integrated workflow automation——这些产品都没有在"持续运转的团队"这个层次上建立产品抽象。Digital Company 如果能真正落地这个差异，确实有机会占据一个目前空白的位置。

但这里有一个需要正视的风险：**"持续运转"这个承诺，在 LLM 能力边界内是否真的可兑现**。跨周期记忆、组织状态感知、绩效差异——这些在技术文档里都有对应设计，但每一条在第一版都会遇到 LLM 一致性问题。用户如果在第二周期发现团队"失忆"或"人格漂移"，对产品信任的伤害是决定性的。产品定义应该在成功判定标准里明确区分"技术上的持续性"和"用户感知上的持续性"，后者才是真正的验收标准。

### **目标用户画像：准确，但边界还需要一刀**

文档把目标用户描述为"一人公司创始人、独立顾问、小型 agency 主理人"，这个方向是对的。但这几类用户的关键差异被模糊了。一人公司创始人和独立顾问对"我理解它为什么这样做"的诉求非常强，他们愿意投入时间 onboard 一支数字团队；而小型 agency 主理人则更在意交付物能不能直接给客户看，他们对"品牌声音一致性"的要求会更严格。**第一版建议只锁定其中一类**，而不是三类并列。原因是这三类用户对"老板接口"的期望形态差异明显，会导致 Briefing 和 Decision 的 UI/UX 设计无法收敛。

### **"它不是什么"的部分：写得好，但少了一个**

文档里列的那些"不是"都很准确，但漏掉了一个当前最危险的竞品滑向：**"一个自动内容发布工具的高级包装"**。如果产品在用户心智里变成"帮我自动发内容的工具"，它就失去了"经营团队"这个核心价值主张，也失去了让用户愿意持续付费的理由。这个边界值得在文档里明确说一句。

### **"成功判定标准"：需要量化**

第十五节的成功判定句写得很有力，但全部是定性描述。对于一个要进入 Phase 0 Spike 的产品，建议补充一个最小可测量的验收指标，例如：用户在第二周期内是否主动回来审批交付物，而不需要被提醒。这比"用户觉得团队熟悉我的业务"更容易在早期验证。

## 二、技术设计评审

### **架构选型：Next.js + LangGraph.js + PostgreSQL + Inngest**

这个组合在当前（2026 年初）已经是有真实生产案例支撑的选型。Inngest 的客户里已经有 Day.ai（AI-driven CRM）、Windmill（AI agent 管理员工）、Mega SEO（AI 内容生产）等与 Digital Company 场景高度相似的产品，而且 Inngest 明确强调自己是"AI workflows 的 nervous system"，这和文档里对 Scheduler 的定位完全吻合。[Inngest](https://www.inngest.com/customers)

关于 LangGraph.js 的成熟度问题，GitHub 上确实有开发者在 2025 年 2 月提问"Is LangGraph.js production ready"，官方的回应是功能上与 Python 版本已基本对齐，但 Python 生态的第三方集成（尤其是研究类工具，如 Tavily、Exa 等）仍然更丰富。[GitHub LangChain](https://github.com/langchain-ai/langgraphjs/issues/850) 技术文档里的 ADR-001 对这个代价是诚实的（"Python 生态里某些研究能力接入稍晚"），但这个代价在内容增长场景里可能比预期更高——**研究员角色的核心能力就是调用搜索和信息聚合工具**，如果这部分工具在 JS 侧缺失，会直接影响第一版最核心的交付物质量。

**建议**：在 Phase 0 Spike 阶段，明确测试研究员 agent 在 LangGraph.js 下能接入哪些信息源（Tavily、Exa、Perplexity API 等都有 REST API，可以绕过 Python SDK），并把这个结论写进 ADR-001 的补充说明里。如果发现缺口太大，可以考虑研究员 graph 单独用 Python FastAPI 实现，其余保持 TypeScript 统一栈。

### **四条核心 Graph 的设计：结构合理，但 Graph B 过重**

Graph A（Cycle Planning）、Graph C（Chief of Staff Briefing）、Graph D（Review & Feedback）的边界都很清晰。**Graph B（Execution Graph）承接了研究、写作、编辑、分发准备四个阶段，这是整个系统里最复杂的部分，但文档对它的描述最简略**。在实际实现中，这四个阶段的 LLM 调用密度、失败模式和人工干预点差异极大。建议把 Graph B 拆分为至少两个子图：一个 Research Graph（信息收集与摘要），一个 Production Graph（写作、编辑、版本管理）。这样做的好处是两个子图可以独立调试、独立定价（研究调用便宜模型，写作调用高质量模型），也更容易在 checkpointer 层面追踪状态。

### **秘书长层（Chief of Staff）：产品最有辨识度的部分，但技术设计最薄弱**

第八节的设计原则写得很好（"不直接暴露原始 agent 对话"、"所有升级都要可解释"），但 Graph C 在技术文档里只有寥寥几行。秘书长层本质上是一个**事件聚合 + 优先级判断 + 压缩摘要**的 pipeline，它的输入是非结构化的内部事件流，输出是结构化的 Briefing 对象。这里有几个技术细节需要在 Spike 阶段验证：

第一，**升级阈值的配置化**。文档里提到"高优项目阻塞超过 N 小时"，但 N 是谁来设置、怎么学习的？第一版可以硬编码，但需要明确写出来，避免后来被遗忘。

第二，**Briefing 的去重逻辑**。同一个阻塞事件可能在多个 daily tick 里被重复触发，秘书长层需要有幂等性保证，避免老板每天收到同一条风险提示。这在技术文档里完全没有提及。

第三，**Briefing 与 Decision 的关系**。文档里 Briefing 和 Decision 是两个独立对象，但在用户侧它们应该是同一个"收件箱条目"。建议在数据模型层面明确 Briefing 到 Decision 的 promotion 逻辑，否则前端很难设计。

### **记忆层设计：轻量策略是对的，但分类需要更细**

ADR-003 选择不上重型 graph memory，这个决策在 MVP 阶段是正确的。但文档里的记忆分类（品牌约束、高表现主题、常见返工点、已做过的关键判断、用户编辑偏好）混合了两种不同性质的记忆：**规则性记忆**（品牌约束、用户偏好）和**经验性记忆**（高表现主题、返工点）。前者应该是结构化字段，直接存在 Team 表里，每次 graph 启动时注入；后者才是语义检索的对象，存在 pgvector 里。如果两类混在一起用同一套检索逻辑，会导致规则性约束被稀释（比如"不用竞品名称"这条规则如果只靠语义检索来召回，有概率被遗漏）。

### **成本控制：ADR 里提到了模型分层，但缺少量化估算**

文档里的成本缓解措施是"低风险步骤优先用便宜模型"，这是对的，但对于一个按周期运转的内容团队，一个周期的 token 消耗量级是多少，文档里没有任何估算。这不是吹毛求疵——内容增长场景的 token 消耗可能非常高（一篇长文初稿 + 研究摘要 + 若干社媒文案 + daily briefing × 7 天），如果不在 Phase 0 做一次 token budget 估算，定价模型就无法建立，产品就无法判断 unit economics 是否成立。

## 三、竞品格局补充

从当前市场来看，与 Digital Company 最接近的方向是以下几类，但都没有真正建立"持续运转团队"这个抽象：

**CrewAI** 是 role-based 多 agent 框架，有 crew 的概念，但没有跨周期记忆，也没有 boss 接口层，本质上还是 workflow。**Lindy.ai** 是 no-code AI agent 平台，有一定的任务持续性，但对象是单个 agent 而不是团队，也没有组织状态管理。**11x.ai** 在 Inngest 的客户列表里出现，主打 AI SDR（销售代表），是垂直方向的数字员工产品，但聚焦销售场景，没有内容团队的组织抽象。**Mega SEO** 也在 Inngest 客户列表里，是 AI 驱动的内容生产产品，但定位是工具而非团队。

这意味着 Digital Company 在产品定义层面目前没有直接对手，但也意味着需要自己教育用户什么是"经营一支数字团队"——这个教育成本不可低估，应该在 GTM 策略里提前规划。

## 四、需要补充进文档的内容

技术文档 v0.1 目前缺少以下几个部分，建议在 v0.2 里补充：

第一，**Token Budget 估算**：一个典型周期的模型调用次数和 token 消耗量级，以及对应的成本区间。这是定价模型的基础。

第二，**Graph B 的详细拆分**：Research Graph 和 Production Graph 的节点设计、状态流转和 HITL 点。

第三，**Briefing 幂等性设计**：同一事件在多个 tick 里的去重逻辑。

第四，**记忆层的分类存储策略**：规则性记忆（结构化字段）vs 经验性记忆（pgvector）的分离设计。

第五，**Phase 0 Spike 的验收标准**：四个 Spike 各自的 done criteria，而不只是描述要做什么。

---

## 总体判断

这两份文档合在一起，已经达到了一个可以开始 Phase 0 Spike 的质量门槛。产品定义的自我克制是最大的优点，技术选型的合理性有真实案例背书。主要风险集中在三个地方：LangGraph.js 在研究工具集成上的能力边界、Graph B 的复杂度被低估、以及"持续运转"承诺在用户感知层面的可信度建立。这三个问题都可以在 Phase 0 Spike 阶段得到答案，不需要等到 Phase 1 才面对。
