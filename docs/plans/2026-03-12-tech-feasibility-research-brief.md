# `Digital Company` 技术可行性调研输入稿

日期：2026-03-12  
状态：Ready for Feasibility Research

## 一、当前判断

当前这版产品定义：

- **不适合作为对外终稿**
- **适合作为内部产品定义冻结版，进入技术可行性调研**

原因很简单：

它已经足够清楚地回答了以下问题：

- 第一版做什么
- 第一版不做什么
- 用户买的是什么
- 产品对象到底是“公司”还是“团队”
- 用户主要通过什么接口使用它

但它还没有完全回答以下问题：

- 精确 ICP 与定价
- 服务边界
- 反馈闭环深度
- 对外 go-to-market 叙事

所以，现在最合适的状态不是继续把它抛来抛去做概念打磨，  
而是：

## `以当前 v0.2 作为产品定义输入，启动技术可行性调研；同时并行继续做产品定义 v0.3 收敛。`

---

## 二、为什么现在可以进入技术调研

因为技术调研真正需要的是一个足够稳定的“最小产品北极星”，而不是一份完美商业计划。

当前版本已经有稳定北极星：

### 第一版产品对象

不是完整公司，而是：

**一支持续运转的数字内容增长团队 / 数字内容增长部**

### 第一版用户关系

用户不是 orchestration operator，而是：

**接手并经营这支团队的业务 owner**

### 第一版价值锚点

用户每个周期拿到两类东西：

- 业务资产
- 管理材料

### 第一版产品接口

老板接口围绕：

- 简报
- 待拍板事项
- 交付物
- 风险升级
- 周期复盘

这已经足够技术调研围绕真实对象展开，而不会再偏到：

- 多 Agent 工具
- 模拟游戏
- 开放世界沙盘

---

## 三、但它还不能算“最终终稿”

我建议把现在这版定义理解成：

## `产品定义冻结版 v0.2`

不是：

## `对外终稿 v1.0`

也就是说，它已经足够拿来做：

- 技术可行性调研
- 架构拆解
- 系统原型
- MVP 技术取舍

但还不适合直接拿来做：

- 正式官网文案
- 销售材料
- 定价页
- 融资叙事

差别就在于：

### 内部冻结版回答的是“我们到底要做什么”

### 对外终稿回答的是“市场为什么现在会买”

前者我们现在基本够了，  
后者我们还没有完全打透。

---

## 四、UI 确实应该跟着重构

这是一个必须同步调整的点，不是可选优化。

因为产品定义已经发生了关键变化：

- 从“经营一个数字公司”
- 收缩成“接手一支持续运转的小团队 / 数字部门”

这会直接影响 UI 的信息对象和语义。

### UI 层需要同步变化的地方

#### 1. 首页主语要变

不再是：

- 公司在运行
- 公司 KPI
- 公司组织

而要变成：

- 团队本周期在推进什么
- 这支团队现在交付了什么
- 这支团队需要老板判断什么

#### 2. 首页的“经营感”要变轻

不再强调：

- 公司级模拟
- 组织世界感
- 公司治理隐喻

而要强调：

- 团队节奏
- 交付流水线
- 秘书长编译层
- 资产沉淀

#### 3. 团队页要更像“部门运营”

不是“组织宇宙”，而是：

- 团队编制
- 岗位分工
- 负载情况
- 候选人补强
- 质量与节奏

#### 4. 首页词汇要系统性调整

例如应优先替换成：

- 公司 -> 团队 / 内容增长部
- 公司脉搏 -> 团队脉搏
- 经营控制台 -> 团队工作台 / Boss Workbench
- 公司资产 -> 内容资产 / 团队资产

所以，UI 不是小修小补，  
而是一次“对象层重构”。

## 五、技术调研应该回答什么

下面这些问题，是现在最值得做的技术调研项。

### 1. 团队长期记忆怎么存

要调研：

- 业务背景记忆
- 岗位级记忆
- 员工级记忆
- 项目级记忆
- 历史交付物记忆

关键不是“怎么存文本”，而是：

**怎么让第二周的团队真的比第一周更像一个熟悉用户业务的团队。**

### 2. 周期推进引擎怎么实现

要调研：

- 周期对象如何建模
- 每周目标如何拆任务
- 每天推进如何触发
- 什么情况下自动推进
- 什么情况下升级给老板

关键是：

**产品是 cadence-driven，而不是 prompt-driven。**

### 3. 秘书长编译层怎么做

要调研：

- 如何把内部事件压缩成老板简报
- 如何形成会议纪要
- 如何生成决策包
- 如何定义升级阈值

这层是产品辨识度很高的部分，不能只靠 UI 假装存在。

### 4. 交付资产流水线怎么做

要调研：

- brief -> research -> draft -> edit -> distribute 的状态流
- 版本管理
- 审核门禁
- 资产归档与复用

关键是：

**资产要成为一等对象，不是聊天的副产物。**

### 5. 反馈闭环怎么做最小版

要调研：

- 发布状态是否入库
- 用户是否采纳资产
- 哪些资产进入下一周期复用
- 简单表现反馈是否能回流策略层

第一版不一定要全自动接平台，但必须有最小闭环。

### 6. 同岗位多人如何落地

要调研：

- 如何实现部分岗位多员工
- 如何表达同岗差异
- 如何做竞争与筛选
- 如何可视化绩效而不游戏化

### 7. 成本模型能不能跑通

要调研：

- 一个团队一个周期需要多少模型调用
- 哪些步骤可降本
- 哪些步骤必须高质量模型
- 秘书层和审核层的 token 成本占比

因为这个产品天然比单 agent 工具更重，  
成本模型必须尽早验证。

### 8. 集成边界怎么划

要调研：

- 第一步是否接 CMS
- 是否接 Notion / Feishu / Docs
- 是否接公众号 / 小红书发布链路
- 是否只做导出，不做直接发布

这个问题会直接影响 MVP 范围。

## 六、我建议的技术调研顺序

不要把所有问题并行打开。  
建议按下面顺序推进：

### 第一阶段：先验证最小闭环是否成立

目标：

- 能创建团队
- 能围绕周期目标推进
- 能产出资产
- 能产出老板简报
- 能让老板拍板

### 第二阶段：再验证“持续性”是否成立

目标：

- 跨周期记忆
- 资产复用
- 团队熟悉度提升
- 基础反馈回流

### 第三阶段：再验证“组织感”是否成立

目标：

- 同岗多人
- 绩效差异
- 招聘补强
- 组织状态可见

这个顺序很重要。  
不要一开始就试图证明“数字组织全部成立”，  
先证明“这支小团队真的能持续交付”。

---

## 七、建议立刻开启的 3 个技术 Spike

### Spike A：周期驱动团队原型

目标：

- 给一个业务画像和周期目标
- 自动生成团队任务推进
- 输出第一轮资产与简报

### Spike B：秘书长编译原型

目标：

- 输入一组内部事件
- 输出老板可读的：
  - 今日简报
  - 会议纪要
  - 待决策事项

### Spike C：资产流水线原型

目标：

- 让内容资产从 research 走到 brief、draft、review、archive
- 验证交付物是否能成为系统核心对象

## 八、最终判断

如果你的问题是：

### `这版能不能算产品定义终稿？`

我的答案是：

**不能算对外终稿，但可以算内部冻结版。**

如果你的问题是：

### `这版能不能进入技术可行性调研？`

我的答案是：

**可以，而且现在就该进入。**

更准确地说：

## `产品定义 v0.2 足够支持技术调研启动；UI 应同步重构；商业与定位定义继续并行打磨。`
