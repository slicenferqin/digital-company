# Boss 工作台 UI 方案（第一版）

日期：2026-03-11
状态：Draft v1

## 1. 产品交付给用户的到底是什么

我对这个问题的结论很明确：

## 不是一个“看热闹的公司模拟器”，而是一个 `Boss Operating System`

也可以中文化地叫：

- `BOSS 工作台`
- `老板驾驶舱`
- `经营控制台`

但产品本质是同一个：

**它交付给用户的是一个可以经营数字公司的操作界面。**

用户打开后要完成的不是聊天，而是五类老板动作：

1. 看经营状态  
2. 看秘书长整理好的简报  
3. 处理关键决策  
4. 审核真实交付物  
5. 调整组织与用人

所以，答案是：

## `是，一个 BOSS 工作台，但不能只是 dashboard。`

它必须同时具备：

- 总览能力
- 决策能力
- 审批能力
- 组织管理能力

如果只有指标和图表，那只是 dashboard。  
如果只有待办和通知，那只是 inbox。  
如果只有团队列表，那只是 admin panel。

真正对的产品形态应该是：

## `Overview + Briefing + Inbox + Artifact Review + Team/Hiring`

## 2. 第一版交付形态建议

第一版我建议用一个主壳层，下面 5 个核心工作区。

### 工作区 1：Boss Workbench / 经营总览

这是用户每天进入的第一页。

要回答四个问题：

- 公司今天在发生什么
- 本周目标离完成还有多远
- 现在有哪些阻塞
- 老板下一步最该做什么

这个页面不是用来细看内容，而是用来“建立经营感”和“锁定注意力”。

### 工作区 2：Chief of Staff Briefing / 秘书简报

这是承上启下最关键的一层。

它不是消息流，而是秘书长整理后的老板视角汇报：

- 今日摘要
- 会议纪要
- 风险升级
- 关键变化
- 推荐决策

这个页面的意义是：

**老板不用参加所有会议，但必须看见被正确编译过的信息。**

### 工作区 3：Manager Inbox / 管理者收件箱

这是老板真正介入公司的地方。

它只收关键事项：

- 优先级冲突
- 方向分歧
- 招聘申请
- 关键稿件审批

每条事项都必须带：

- 背景
- 推荐方案
- 风险
- 影响范围

老板在这里做的是决策，不是聊天。

### 工作区 4：Artifact Center / 交付物中心

这是价值兑现层。

用户在这里看到的必须是：

- 文章
- brief
- 研究摘要
- 社媒内容
- 周报

用户必须能对这些交付物做：

- 通过
- 打回
- 批注
- 设为模板

### 工作区 5：Team & Hiring / 团队与招聘

这是经营感最强的地方。

它让用户感受到：

- 团队成员不是一次性生成物
- 每个人表现不同
- 组织有扩编、替换、试用和成长

这个页面至少要同时覆盖：

- 当前编制
- 绩效差异
- 负载情况
- 招聘申请
- 候选人 shortlist

## 3. 为什么不是“老板聊天页”

很多 Agent 产品很容易滑向一个错误方向：

- 左边一排角色
- 中间一个大聊天框
- 大家轮流说话

这类界面的问题是：

1. 老板不像老板，像群聊管理员  
2. 交付物被过程淹没  
3. 组织状态不可见  
4. 决策入口不清楚

你这个产品真正要卖的是：

**“这家公司在替我运转，我只在关键时刻做判断。”**

所以 UI 也必须服务这个承诺。

聊天最多是证据层、日志层、诊断层，不能是主操作层。

而秘书简报层要承担一个额外职责：

**把“组织发生了什么”压缩成“老板需要知道什么”。**

## 4. 第一版首屏应该长什么样

首屏我建议采用：

## `顶部摘要带 + 左侧导航 + 中央经营视图 + 右侧老板动作区`

### 顶部摘要带

展示：

- 公司名
- 本周目标
- 预算剩余
- 待审批数量
- 阻塞项目数
- 主 CTA

### 左侧导航

放稳定导航，不做花哨世界地图：

- 总览
- 秘书简报
- 管理者收件箱
- 项目与交付物
- 团队与绩效
- 招聘与扩编
- 组织时间线
- 制度设置

### 中央经营视图

这是首屏的主阅读区，应放：

- 北极星摘要
- 秘书今日简报
- 今日事件流
- 新交付物摘要
- 主要风险

### 右侧老板动作区

这块只放“现在该老板处理什么”：

- 秘书长推荐处理顺序
- 待批准事项
- 需要拍板的方案
- 高优风险提醒

这样首屏会非常像“驾驶舱”，而不是项目管理软件。

## 5. 我给这版 UI 的一句定义

### `老板不是来操控 Agent 的，而是来经营一家正在运转的数字公司。`

所以 UI 的目标不是展示多 Agent 有多聪明，  
而是让用户在 30 秒内建立这种感受：

- 公司已经在跑
- 有人替我过滤信息
- 我只处理关键节点
- 新资产正在被交付
- 组织可以被我塑造

## 6. 降低心智压力的布局原则

这次我把页面继续收缩时，采用的是下面 4 条硬原则：

### 原则 1：先摘要，后细节

首页不要先给老板看全部事件、全部会议、全部指标。  
应该先给：

- 今天只看哪三件事
- 今天不用看什么
- 今天先做哪个判断

也就是让用户先获得“秩序感”，再决定是否下钻。

### 原则 2：一屏只保留一个主叙事

首页的主叙事应该是：

**“今天这家公司已经替你推进了什么，而你现在只需要做什么。”**

如果一屏同时承载：

- 时间线
- KPI
- 决策
- 招聘
- 交付物
- 团队细节

那老板会立刻进入分心状态。

### 原则 3：过程信息后移到秘书层

所有过程性信息都不该先出现在首页：

- 会议过程
- 角色争论
- 长日志
- 多轮评审细节

这些都应该先经过 `秘书长 / Chief of Staff` 编译，变成：

- 简报
- 纪要
- 决策包
- 风险升级单

### 原则 4：明确告诉老板“你现在不用看什么”

这是经营 UI 很容易忽略的一点。

减少压力，不只是减少内容量，  
而是明确帮用户判断：

- 哪些事先别管
- 哪些事已被团队接住
- 哪些事以后再处理也没关系

这个体验会非常像一个成熟的老板办公室，而不是一个吵闹的 Agent 房间。
